Architecture review

Freeflow activation contract interface designs

Generated from four explorer subagents. Candidate: deepen the always-on activation contract so setup, docs, fixtures, and eval assertions stop copying drift-prone text.

Problem space: the compact Freeflow activation block and host setup rules live in setup skill prose, runtime docs, research notes, setup fixtures, and registry assertions. One edit can drift across Plugin Runtime evidence.

Design 1: three-entry contract

Most precise 
getActivationContract()
planActivation()
renderActivation()
Interface
Contract, plan, render. Targets include Codex, Claude core, Claude import, fixtures, eval assertions, docs.
Implementation hides
canonical text, config schema, host matrix, hard stops, placement rules, eval snippets.
Trade-off
High depth; risk is becoming a broad doc generator.

Design 2: flexible activation planner

Most extensible 
resolveActivationPlan(params) -> ActivationPlan | ActivationStop
Interface
Repo state, request intent, host adapters, core contract.
Implementation hides
host details, source-truth conflicts, fixture validation shape.
Trade-off
Future-host friendly, but less explicit about render targets.

Design 3: common setup interface

Best default path 
activateFreeflow(params): ActivationResult
Interface
repoRoot, requestedHost, defaultMode, existing files, repo instruction text.
Implementation hides
host detection, conflict scanning, block rendering, config parse/write, verification.
Trade-off
Great setup ergonomics; weaker for docs/eval assertion drift.

Design 4: host ports and drift checks

Best seam discipline 
planFreeflowActivation({ hostTarget, requestedDefaultMode, hostSnapshot, canonicalCoreContract })
Interface
Activation plan or stop with canonical core contract and host snapshot.
Adapters
CodexInstruction, ClaudeInstruction, Config, FixtureEval, DocsExcerpt.
Trade-off
Strong drift control; slightly more adapter ceremony.

Comparison

Design Depth Locality Thin spot
Three-entry contractHigh: contract/plan/renderHigh: one canonical textCould generate too much docs text
Flexible plannerMedium-highHigh for host setupAdapter surface broadens quickly
Common setupHigh for setup-freeflowMedium for docs/evalsDoes not fully solve fixture/docs drift
Ports/drift checksHigh at host seamHigh: drift checks explicitMore structure before implementation

Recommendation

Hybrid: canonical contract plus host adapters and drift checks

Use Design 1's contract/plan/render shape, Design 4's Codex/Claude adapters, and Design 3's simple setup call. Keep docs/eval outputs as checks or excerpts, not a broad generator.

setup-freeflow
  -> planActivation(contract, repoState, request)
       -> CodexActivationAdapter | ClaudeActivationAdapter
       -> ConfigAdapter
       -> Drift checks for fixtures/docs/eval assertions
