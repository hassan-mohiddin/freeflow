Interface review

Skill Evidence Grouping

Freeflow evidence is currently split by file kind: eval registries, prompts, fixtures, command surface, suites, reports, and generated runs. The deepening opportunity is a small interface that answers: "what evidence backs this skill?" without making callers know the repo layout.

Problem Space

Constraints

  • Registries remain source truth for eval definitions.
  • Command surface JSON remains source truth for command routing.
  • Reports are result evidence, not definition authority.
  • Generated runs stay private and ignored unless explicitly requested.
  • Acceptance evidence must prefer fresh acceptance runs over old smoke reports.

Dependencies

  • Data dependencies: JSON registries and command surface.
  • Document dependencies: Markdown reports, suites, and matrices.
  • Runtime dependencies: existing fixture runners and host adapters.
  • Policy dependency: public package must not leak generated run output.

Sketch

skill-evidence.sh verify-work
  -> getSkillEvidence({ skill: "verify-work" })
     -> registries
     -> command-surface
     -> suites
     -> reports
     -> run policy

Current Shape

Before: shallow locality

flowchart TD Caller[Maintainer asks: evidence for verify-work?] Caller --> R1[fixture-evals.json] Caller --> R2[verify-work-evals.json] Caller --> R3[command-surface.json] Caller --> R4[command matrix] Caller --> R5[acceptance suite] Caller --> R6[reports/by-skill] Caller --> R7[runs ignored/private]

Design 1: Minimal SkillEvidence Module

Strong

Two entry points: get evidence for one skill, or validate evidence grouping. This maximizes leverage for the most common maintainer question while keeping public output separate from internal generated runs. 

getSkillEvidence({
  skill: "review-work",
  depth?: "shallow" | "deep",
  audience?: "public" | "internal",
  includeHistorical?: boolean
})

validateSkillEvidence({ skill?: string, audience?: "public" | "internal" })

Leverage

Callers get one bundle: skill file, commands, acceptance rows, eval definitions, latest reports, and private run refs.

Locality

The filesystem layout stays put; evidence joins concentrate in one module.

Thin edge

Markdown table parsing remains brittle unless later replaced with explicit metadata.

Design 2: Common Caller CLI + Skill Index

Strong

A developer-facing command answers the daily workflow: "I changed this skill; show me evidence and commands." A declarative skill evidence index owns only grouping metadata. 

plugins/freeflow/evals/scripts/skill-evidence.sh <skill> \
  [--format summary|json|commands] \
  [--run-label LABEL] \
  [--agent codex|claude]

plugins/freeflow/evals/registries/skill-evidence.json

Leverage

Lowest-friction interface for maintainers and eval authors.

Locality

Explicit grouping avoids brittle ID-prefix inference.

Thin edge

An index can drift unless validation makes missing paths and stale links fail loudly.

Design 3: Flexible EvidenceGroup Query

Worth exploring

A broader query interface supports skill, command, suite, or eval subjects. It is more extensible but risks becoming a search layer unless the source precedence rules stay sharp. 

getEvidenceGroup({
  subject: { kind: "skill", name: "verify-work" },
  include: ["evalDefinitions", "prompts", "fixtures", "commandSurface", "suites", "reports"],
  variants: ["baseline", "with-skill"],
  freshness: "acceptance-over-smoke"
})

Leverage

One module could power dashboards, reports, dry-runs, and audits.

Locality

Normalizes evidence by subject instead of by filesystem directory.

Thin edge

The interface surface is wider than needed for the first caller.

Design 4: Ports and Adapters Evidence Graph

Strong

This design makes the source boundaries explicit. It matters because Freeflow already has multiple eval registries: fixture evals, verify-work evals, and handoff evals. 

type EvidenceQuery = {
  skill?: string
  command?: string
  evalIds?: string[]
  suite?: string
  include?: Array<"registry" | "prompts" | "fixtures" | "reports" | "suites" | "commandSurface">
  freshness?: "current" | "latest-report" | "all"
}

Leverage

Handles multiple registry shapes without making callers know which registry owns which eval family.

Locality

Adapters isolate JSON, Markdown, command, report, and run-policy parsing.

Thin edge

More architecture than a single helper script; should be introduced around concrete callers.

Comparison

Design Depth Locality Risk
Minimal SkillEvidence High for skill-focused callers. Strong: one bundle per skill. May under-serve command and suite queries.
CLI + Skill Index High for daily eval work. Strong if index is validated. Index drift without validation.
EvidenceGroup Query Broad but less focused. Strong across subject types. Can become a generic search interface.
Ports and Adapters High behind a small public facade. Strongest source isolation. Needs discipline to avoid overbuilding.

Recommendation

Use a hybrid: a common-caller CLI over a small queryable SkillEvidence module, backed by source-specific adapters. Add a small explicit evidence index only for ownership/grouping metadata that cannot be safely derived from source truth. 

plugins/freeflow/evals/scripts/skill-evidence.sh <skill> [--format summary|json|commands]
  -> getSkillEvidence({
       subject: { kind: "skill", name: skill },
       freshness: "acceptance-over-smoke",
       audience: "public"
     })
     -> RegistryAdapter(fixture, verify-work, handoff, future)
     -> CommandSurfaceAdapter(command-surface.json as authority)
     -> SuiteAdapter(acceptance gates)
     -> ReportAdapter(results, newest first)
     -> RunPolicyAdapter(public/internal)

Why this is strongest

It gives the maintainer a trivial interface while placing the real depth where Freeflow needs it: joining multiple evidence sources without copying stale inventories into docs.

Guardrails

Do not move prompts, fixtures, or reports into per-skill directories yet. Do not treat reports as canonical. Do not leak generated runs into public plugin artifacts. Validate every source link.
