Interface review

Release Metadata Validation

Freeflow release truth is split across root marketplace indexes, plugin manifests, command-surface metadata, package-boundary decisions, release evidence, and deferred smoke-test notes. The deepening opportunity is one validation module that finds drift without creating a generated runtime mirror or adding v0.1 enforcement.

Problem Space

Constraints

  • the repo root remains the only runtime source of truth.
  • Root Codex and Claude marketplace indexes must point at that runtime.
  • Codex and Claude manifests must agree on stable identity.
  • `nativeSlashHandlers=false` means no manifest command handlers.
  • Generated eval runs stay ignored and outside release truth.
  • Install smoke checks remain optional/deferred unless release mode asks for them.

Dependencies

  • Metadata: marketplace indexes and plugin manifests.
  • Registry: command-surface JSON and matrix report.
  • Docs: architecture, release evidence, release boundary ADR.
  • Git/package: ignored paths, old-identity scans, duplicate package checks.
  • Host smoke: Codex/GitHub/Claude adapters later.

Sketch

validateReleaseMetadata()
  -> discover metadata
  -> normalize release model
  -> check identity/locality
  -> check command surface
  -> check boundary/docs
  -> report findings

Design 1: Minimal Validator

Strong

One entry point validates v0.1 release metadata and returns a focused report. Existing `audit-command-surface.sh` can be wrapped instead of rewritten immediately. 

validate({
  repoRoot: process.cwd(),
  profile: "v0.1",
  strict: true
}) -> ValidationReport

Leverage

High for local prepublish checks with little caller knowledge.

Locality

Concentrates identity, locality, command-surface, and boundary rules.

Thin edge

Does not prove external GitHub/Claude install behavior.

Design 2: Flexible Profiles

Worth exploring

Profiles model local release, package cleanliness, docs drift, GitHub install smoke, Claude smoke, and native slash readiness. Strong future fit, but broad for the first seam. 

validate({
  repoRoot,
  pluginId: "freeflow",
  runtimeRoot: "the repo root",
  profiles: ["local-release", "package-cleanliness", "docs-drift"],
  hosts: ["codex", "claude", "local"]
})

Leverage

Covers future smoke and native handler checks without changing interface.

Locality

Keeps validation rules modular by profile.

Thin edge

Can become a release framework if every concern becomes a profile.

Design 3: Common Maintainer Call

Strong

Optimize for the person preparing a release: call one function or script from repo root and get failures, warnings, and evidence. 

validateReleaseMetadata() -> {
  status,
  version,
  checks,
  failures,
  warnings,
  evidence
}

Leverage

Best day-one ergonomics and easiest to put in prepublish docs.

Locality

Default discovery hides all current path details.

Thin edge

Needs escape hatches for future smoke profiles.

Design 4: Ports and Adapters

Strong

Host formats, docs parsing, package scans, and future install smoke checks are adapters behind one normalized release model. 

validate({
  repoRoot,
  pluginRoot: "the repo root",
  releaseVersion: "0.1.0",
  mode: "prepublish",
  adapters: {
    codexMarketplace,
    claudeMarketplace,
    codexManifest,
    claudeManifest,
    commandSurface,
    docs,
    gitPackage,
    installSmoke?
  }
})

Leverage

Best isolation for Codex, Claude, docs, git, and future host volatility.

Locality

Release rules live in the validator; raw host formats stay behind adapters.

Thin edge

Can feel heavy unless wrapped by a trivial default caller.

Comparison

Design Depth Locality Risk
Minimal Validator High for v0.1 deterministic checks. Strong: one release model. Weak on future smoke profiles.
Flexible Profiles Broad, extensible depth. Strong if profiles stay bounded. May become too general.
Common Maintainer Call High day-one leverage. Strong for prepublish flow. Needs adapters hidden below.
Ports and Adapters Strongest implementation seam. Strongest source isolation. Heavy without default wrapper.

Recommendation

Use a hybrid: a trivial `validateReleaseMetadata()` default over a ports-and-adapters validator with bounded modes: `local`, `prepublish`, and `release`. Add optional profiles only for checks with different cost or environment needs. 

validateReleaseMetadata()
  -> ReleaseMetadataValidator.validate({
       pluginRoot: "the repo root",
       mode: "prepublish",
       checks: defaultLocalChecks
     })
     -> HostMetadataAdapters(Codex, Claude)
     -> CommandSurfaceAdapter(existing audit first)
     -> DocsEvidenceAdapter
     -> GitPackageScanAdapter
     -> Optional InstallSmokeAdapter

Why this is strongest

It gives maintainers one command-sized interface while concentrating release metadata rules in one module. Host volatility and optional smoke checks stay behind adapters.

Guardrails

Do not add hooks, native slash handlers, or a generated package copy. Do not mechanically over-parse docs. Treat install smoke as optional until a release mode explicitly requires it.
