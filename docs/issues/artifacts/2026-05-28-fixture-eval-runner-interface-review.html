Architecture review

Freeflow fixture eval runner interface designs

Generated from four explorer subagents after explicit approval. Candidate: deepen the Adversarial Fixture Eval runner module while keeping host commands as adapters.

Problem space: current scripts repeat fixture staging, prompt assembly, host invocation, output capture, diff capture, git metadata, dry-run behavior, and policy checks across Codex and Claude paths.

Design 1: minimal single entry point

Deepest interface 
runFixtureEval(request) -> result
Interface
evalId, variant, agent, runDir, outputFile, skillFiles, registryFile, dryRun, requireEmptyDiff, hostOptions.
Implementation hides
registry lookup, prompt materialization, fixture staging, host command shape, sidecar output naming.
Trade-off
High locality. Shell may stay shallow for structured validation.

Design 2: flexible typed request plus dependency ports

Most extensible 
runFixtureEval(request: FixtureEvalRunRequest, deps: FixtureEvalDeps): FixtureEvalRunResult
Interface
Allows prompt overrides, fixture overrides, setup policy, workspace policy, artifact policy, failure policy, agent options.
Implementation hides
registry, filesystem, shell, artifact store, and agent adapter details.
Trade-off
Flexible but risks a shallow interface if arbitrary variants and overrides become normal.

Design 3: common-caller CLI

Best ergonomics 
plugins/freeflow/evals/scripts/run-fixture-eval.sh <eval-id> [options]
Interface
Defaults: Codex, with-skill, standard registry, manual timestamped run path. Flags preserve baseline, Claude, dry-run, explicit output.
Implementation hides
positional arg noise, env var shape, output path construction, skill mapping.
Trade-off
Great caller leverage. Needs internal typed module or shell parsing becomes the implementation.

Design 4: ports and adapters

Best seam discipline 
runAdversarialFixtureEval(params): FixtureEvalResult
Interface
Strict baseline/with-skill, Codex/Claude, typed env/options. Keeps registry resolution and artifact capture domain-local.
Adapters
AgentAdapter for host commands. FixtureWorkspace for local-substitutable workspace, diff, and git capture.
Trade-off
Explicit seam adds structure, but it matches real Codex/Claude divergence.

Comparison

Design Depth Locality Thin spot
MinimalHigh: one request/resultHigh: runner owns lifecycleMay hide too many host options
FlexibleMedium: many knobsMedium-highInterface can become policy surface
Common CLIHigh for daily useHigh if backed by internal moduleShell option parsing creep
Ports/adaptersHigh at host seamHigh: host divergence isolatedMore design upfront

Recommendation

Hybrid: common CLI over a minimal ports-and-adapters module

Use Design 3 for caller ergonomics, backed by Design 1's small request/result interface and Design 4's AgentAdapter seam. Reject Design 2's broad override surface for v0.1 because it weakens depth and invites policy creep.

run-fixture-eval.sh <eval-id> [--baseline] [--agent codex|claude] [--dry-run] [...]
  -> runAdversarialFixtureEval(params)
       -> AgentAdapter(codex|claude)
